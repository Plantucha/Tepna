## 1 · OxyDex entropy caps are a HEAD-slice, not whole-night decimation — *evaluate*
`oxydex-dsp.js computeHREntropy` (~line 1077) bounds its O(N²) cost with `USE = Math.min(n, 1000)` +
`clean.slice(0, USE)` — i.e. it computes SampEn from the **first ~1000 motion-free samples** (≈ the first
~16 min of an 8 h O2Ring night). `computeSpO2Entropy` (~line 1158) does the same with the first 800.
Cost is correctly bounded (that was FOLLOWUPS-II's only ask), but the metric then **characterises only the
start of the night**, not the whole recording — sleep architecture (apnea clusters, REM-dense late-night
periods, arousal rebounds the corpus plants) is heavily back-loaded, so a head-slice can systematically
under-/over-state full-night HR/SpO₂ regularity.

- **Decide:** is the head-slice an intentional "settle-in / first-cycle" window, or an accidental
  consequence of "use first N for speed"? Compare, on the committed overnight corpus, head-slice SampEn
  vs a **deterministic whole-night decimation** (every ⌈n/cap⌉-th motion-free sample → same O(cap²) cost,
  but spanning the entire night) for both HR and SpO₂ entropy.
- **If decimation is materially more representative:** switch `computeHREntropy` + `computeSpO2Entropy` to
  the **decimation cap pattern already used in `pulsedex-dsp.js sampEn`** (deterministic stride, documented,
  return-shape unchanged). This **MOVES the SpO₂/HR SampEn values** → it is NOT metric-inert: the OxyDex
  fixture must be **regenerated** (re-run the app on its inputs + re-export, never hand-edit) and
  `FIXTURE-PROVENANCE.json` re-recorded with the producing bundle's `manifestHash`, then OxyDex re-bundled
  with `BUILD-MANIFEST.json` hand-updated. Gate: `Dex-Test-Suite.html` green → re-bundle OxyDex → GATE A.
- **If the head-slice is fine / preferable:** document it as intentional in-code (a one-line rationale
  comment on each cap) and add a static guard, mirroring the FOLLOWUPS-II group-9 pattern — no re-bundle.
