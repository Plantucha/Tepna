Register `adapters/resmed-edf.js` (CPAP was the fleet's only adapter-less signal type), and retire the `mode` heuristic — it was measuring EPR, not auto-titration.

**§P3 — the ResMed adapter.** CPAP was the one signal type with no registered adapter ("EDF is binary
+ multi-file, so there is no text-stream adapter"), so the SD-card → `edfSets` grouping had to be
hand-rolled in `tools/`. It now registers like the other eight, taking its bytes off the established
`ctx` escape hatch (`ctx.buffers` / `ctx.edfSets`) rather than the text stream. It carries the
**session-grouping rule (§F4)**, whose second clause is load-bearing and non-obvious:

> cluster files within ±60 s of the set's anchor, **and a SECOND file of a type opens a NEW set**.

Without it, a brief mask-off/on inside one minute writes a second EVE/CSL pair and a naive ±60 s
cluster **overwrites the first — silently dropping that set's scored events**. 8 sessions in the
~180-night corpus under-counted exactly this way. The gate now asserts both that the rule keeps both
EVE files *and* that the naive rule provably drops one, so a "simplification" back to it reds.
Orchestrator-only re-bundle (Data Unifier / OverDex): **no `manifestHash`, no fixture moved.**

**§F2/§P4 — the `mode` heuristic.** `mode = pressIqr > 1.0 ? 'APAP' : 'CPAP'` flipped **57 times**
across ~180 real nights, with 27% of nights within ±0.2 cmH₂O of the cut — the threshold landed
*inside* the distribution. It read as a device setting and was not one.

The root cause is a **confound, not a bad constant**: the raw IQR of delivered pressure is dominated
by **EPR**, which drops pressure 1–3 cmH₂O on every *expiration*, and PLD samples at 0.5 Hz — fast
enough to catch those dips. So a fixed-pressure CPAP with EPR=3 shows a raw IQR > 1.0 and was labelled
"APAP". The old cut was reading the EPR setting.

The fix separates them by **timescale**, which is what actually distinguishes them — EPR modulates
within a *breath* (~4 s), auto-titration drifts over *minutes*. `mode` is now called on a per-5-min
P90 pressure envelope (P90 steps over the expiratory dips), and it **refuses to guess**: between the
thresholds it returns `null`. A night is labelled only if **every session agrees** (the stability
guard §F2 demands) — previously the *first* session's label named the whole night.

Both user-facing surfaces that shipped the discredited rule are gone: the unguarded `mode` chip, and
a subtitle that told the user outright *"Auto-titration spread (>1 ⇒ APAP)"*. New `pressureEnvIqr`
metric (registered, `measured`) surfaces the quantity the call is actually made on.

**Honest limit:** the two cut points are set from physiology and agree with the fixtures (real nights
1.2 / 2.2, synthetic EDF 1.08, a fixed machine ~0), but they are **not calibrated against the ~180-night
corpus**, which is not in this repo. The dead-band makes that safe — an unseparable night reads
"unknown", never a wrong device setting. Calibration is the follow-up.

CPAPDex re-bundled (`manifestHash e2392eda2d0a → c0e985875598`); all **5** CPAP fixtures regenerated by
re-running the real modules (`tools/regen-cpap-goldens.mjs`, new) — `mode` feeds `recording.contentId`,
so every CPAP export's content moved. On the real 06-12 night the stability guard now correctly reports
`mode: null` where two sessions disagree.
