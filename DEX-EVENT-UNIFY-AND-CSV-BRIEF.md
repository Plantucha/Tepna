## 0. Prime directive — do not break standalone independence

Each Dex bundles to ONE self-contained `.html` that loads only its own `*-<node>.js`
plus the shared *source* files (`kernel-constants.js`, `ganglior-provenance.js`,
`crossnight-envelope.js`, `metric-registry.js`). There is **no runtime coupling**
between apps; the Integrator and "Load ECGDex JSON" are opt-in consumers of exported
files. **Nothing in this brief may add a cross-node runtime dependency.** A single-sensor
user (O2Ring only) must still run OxyDex alone, unchanged. Every task below is internal
to one node's files.

## 1. Gates (non-negotiable, from CLAUDE.md)

You edit `*-dsp.js` here, so ALL of these fire:

1. **Edit the `.js`, never the bundled `.html`.** Re-bundle via the inliner after changes.
2. **Regression gate** — open `Dex-Test-Suite.html`, wait ~3 s, `#summary` must be all green.
   Run after ANY `*-dsp.js` change AND after re-bundling.
3. **Re-bundle checklist** — a JS change moves the bundle's `manifestHash`. After
   re-bundling you MUST hand-update that app's entry in `BUILD-MANIFEST.json` to the new
   `manifestHash` (read it off `verify-provenance.html`'s manifestHash column). GATE A
   hard-fails on stale/missing.
4. **Provenance gate** — open `verify-provenance.html`, confirm no red verdicts.
   `buildHash` is a coarse fingerprint; trust `manifestHash`. (A lone buildHash/fixture
   red with no code change is suspect — re-run.)
5. **Fixture regen** — these changes alter computed numbers (intentionally, §2). Regenerate
   each affected node's `uploads/*.json` fixtures by **re-running the app on its inputs and
   re-exporting** (never hand-edit), then record the producing bundle's `manifestHash` in
   `FIXTURE-PROVENANCE.json` (GATE B). This needs no extra re-bundle.
6. **Clock Contract** — unchanged and untouched. Do not modify `parseTimestamp` or any
   time handling. If you read timestamps, use the existing `getUTC*` helpers.
7. **Evidence badges** — every surfaced metric keeps its badge; if you change a metric's
   identity, keep its `evidence` grade in the node registry. No new unbadged numbers.

## 2. TASK A — Unify desaturation-event detection (OxyDex)

### Problem (strictly from the code)
`oxydex-dsp.js` re-implements the same "rolling-baseline → desaturation event" loop in at
least five places, each with its own thresholds and closing logic:

| Function | Baseline used | Drop threshold | Close logic |
|---|---|---|---|
| canonical `detectODI` (ODI-4) | `computeCeilingBaselineArr` (p90) | `K.ODI_DROP` (4%) | hysteresis `K.ODI_HYST` |
| `computeDesatSlopes` | `computeBaselineArr` (mean) | `K.ODI_DROP` | hysteresis `K.ODI_HYST` |
| `computeODI1` | `computeBaselineArr` (mean) | `bl − 1` | `> bl − 0.5` |
| `computeRollingMetrics` (post-dip HR) | `computeBaselineArr` (mean) | `K.ODI_DROP` | re-entry on rise |
| `computeHypoxicLoad` | `computeBaselineArr` (mean) | `bl − 3` (ODI-3) | `≥ 10 s` min |
| `computeBreathingIrregularity` | `computeBaselineArr` (mean) | `K.ODI_DROP` | rise |

Two concrete consequences:
- **The v22.36 ceiling-baseline fix only reached `detectODI`.** Every satellite metric
  still re-detects against the OLD trailing-mean baseline, so slope/recovery/clustering/
  breathing-irregularity/post-dip metrics are computed off a different, known-biased event
  set than the headline ODI-4. They silently disagree.
- **Repeated O(n) passes** — each function recomputes a baseline array and re-walks the
  signal. The changelog's recurring ODI bugs (chatter, undercount) live in these forks.

### Required change
Introduce ONE canonical event detector and have every consumer read its output.

1. Add a single function, e.g.:
   ```js
   // Detect desaturation events ONCE against the ceiling baseline (the v22.36 model).
   // Returns a frozen, reusable array; pass dropPct to get ODI-3 (3) or ODI-4 (4) sets,
   // or detect at the smallest threshold once and derive deeper sets by depth filter.
   function detectDesatEvents(rows, opts) // → [{ startIdx, nadirIdx, endIdx,
                                          //      baseline, nadir, depth, durationSec,
                                          //      dipSlope, recSlope }]
   ```
   - Use `computeCeilingBaselineArr` (NOT the mean) so every consumer inherits the ceiling
     fix. Use `DexKernel.K.ODI_DROP` / `K.ODI_HYST` for the ODI-4 set.
   - Compute the baseline array ONCE and pass it in (or memoize on the rows object for the
     duration of `processNight`).
   - Emit per-event fields that today's five functions each recompute: `dipSlope`,
     `recSlope`, `depth`, `durationSec`, `baseline`, `nadir`, and the indices.

2. Rewrite the consumers to **derive from the canonical list**, deleting their private loops:
   - `computeDesatSlopes` → mean/aggregate over `events[].dipSlope` / `.recSlope`, MODL from
     event sample spans, first-/last-half clustering from `.nadirIdx`.
   - `computeBreathingIrregularity` → inter-nadir intervals from `events[].nadirIdx`.
   - `computeRollingMetrics` post-dip HR → use `events[].nadirIdx`.
   - `computeHypoxicLoad` → request the ODI-3-depth subset from the canonical detector
     (filter `depth ≥ 3`) instead of its own `bl − 3` loop.
   - `computeODI1` → either keep as the one deliberate exception (ODI-1 is a 1% detector by
     definition) **or** add a `dropPct` param to the canonical detector and call it with `1`.
     If kept separate, add a one-line comment stating why (different threshold semantics).
   - `detectODI` (ODI-4) → ideally also routes through the canonical detector so there is
     exactly one definition; if risk is too high in one pass, leave `detectODI` as the
     reference implementation and make the canonical detector call into the SAME inner
     primitive. **There must be one event-detection primitive, not two.**

### Outputs WILL move — that is the fix
Satellite metrics recomputed against the ceiling baseline will change on real data. This is
correct. Validate against the synthetic cohort (see `SYNTHETIC-CORPUS-*`, cohort harness)
that:
- ODI-4 itself is unchanged (it already used the ceiling baseline);
- slope/asymmetry/recovery/clustering shift in the expected direction (no longer dragged by
  the mean baseline) and remain physiologically sane on `none`/`mild`/`severe` strata;
- no metric goes null/NaN where it was previously finite.

Then regenerate OxyDex fixtures (§1.5) and update `BUILD-MANIFEST.json` + `FIXTURE-PROVENANCE.json`.

### Done when
- One event-detection primitive in `oxydex-dsp.js`; the five private loops are gone.
- `Dex-Test-Suite.html` all green (add/extend an assertion that the canonical detector and
  any retained `detectODI` agree event-for-event on a fixed input).
- `verify-provenance.html` clean; OxyDex `manifestHash` updated; fixtures regenerated.

## 3. TASK B — Make the human CSV an export-only boundary

### Decision the product owner has confirmed
**JSON is the machine/Integrator format; CSV is the human-readable export.** Both stay.
The fix is NOT to remove CSV — it is to stop using the human CSV as a *re-import*
(round-trip) boundary, because that is where the bug surface lives.

### Problem (strictly from the code)
`readFile` in `oxydex-dsp.js` detects an `OxyDex Night Summary` / `O2Ring Night Summary`
header and calls `parseSummaryCSV` to **reconstruct the full night object** by
string-matching ~80 human labels. The OxyDex changelog is dominated by this one function:
`0`-is-falsy chains that silently dropped `SBII`/`WASO`/`AAI`/`DesSev`/`pRED-3p`, key-trim
(`'  KEY'`) mismatches, and a `spikes` ReferenceError that zeroed EVERY night on import.
It is lossy by construction and re-acquires bugs each time a metric is added.

### Required change (primary path — recommended)
1. **Keep `exportCSV` exactly as-is** — humans still save and read the CSV. Untouched.
2. **Retire `parseSummaryCSV` as an ingest path.** In `readFile`, when a summary-CSV header
   is detected, do NOT reconstruct a night. Instead surface a clear, non-blocking notice via
   the existing parse-warning banner mechanism:
   > "This is a human-readable summary CSV (export-only). To reload a night, use its `.json`
   > export. Raw O2Ring CSVs and `.json`/`.jsonl` still import normally."
3. Reload of analyzed nights goes through the structured `parseJSONL` / single-JSON path,
   which already round-trips losslessly (it carries the full night object).
4. Delete `parseSummaryCSV` and its now-dead helpers (`_flagSev` is shared with `buildFlags`/
   `parseJSONL` — keep it; remove only code unique to `parseSummaryCSV`). Remove its branch
   in `readFile`. Confirm no other caller references it (grep first).
5. Update any on-page copy / `How to Collect Data.html` / reference guide text that tells
   users they can re-load a summary CSV, so guidance reads "keep the `.json` to reload."

### Fallback (ONLY if CSV re-import must remain a supported feature)
If the owner later decides CSV-in must stay, do NOT keep the brittle label parser. Instead
embed a single machine-readable line in the CSV export — a `# DEXJSON:{...}` comment line
carrying the same structured payload `exportJSON` writes — and have the importer parse THAT
one line, ignoring the human rows entirely. This makes the CSV round-trip exactly as robust
as JSON while staying human-readable. (Primary path above is still preferred; this is a
contingency, not the default.)

### Done when
- `exportCSV` output is byte-unchanged (humans unaffected).
- Raw-CSV, `.json`, `.jsonl`, and `.bin` ingest all still work; only summary-CSV *re-import*
  is removed (or hardened via the fallback).
- `parseSummaryCSV` (or its fragile internals) no longer exists; grep is clean.
- `Dex-Test-Suite.html` green; re-bundle + `manifestHash` + provenance updated.
  (No fixture math changes from this task — it only removes an ingest path.)

## 6. OUT OF SCOPE — do NOT attempt in this work (rationale)
These are real but baked into the shared core / inliner; each re-bundles all 8 apps for
preventive (not corrective) payoff. They need their own deliberate projects.

- **Extracting a shared `clock.js`.** `parseTimestamp` is duplicated per node by the Clock
  Contract on purpose. It is byte-identical, well-tested, and stable. Sharing it touches
  every app + integrator + tests. Leave the duplication; do not "fix" it here.
- **De-globalizing page scope (IIFE modules everywhere).** Whole-fleet blast radius, load-
  order semantics, preventive only. Out.
- **Collapsing the provenance machinery** (`buildHash` + `manifestHash` + `BUILD-MANIFEST`
  + `FIXTURE-PROVENANCE`). Requires owning/extending the inliner (see
  GENERATOR-FOLLOWUPS-II §1). Documented, contained, metadata-only. Out.
- **Adding a typed night-object schema / single `validate()`.** Worthwhile but fleet-wide;
  do it after event-unification is ported everywhere, as its own pass.
