Surfaced while executing **Task A** (the OxyDex desaturation-event unification). Task A landed
the canonical `detectDesatEvents(spo2, opts)` primitive in `oxydex-dsp.js` and rewired the 7
enumerated consumers (detectODI, computeDesatSlopes, computeODI1, computeRollingMetrics post-dip,
computeHypoxicLoad, computeBreathingIrregularity, computeSpO2Advanced) onto the ONE trailing-p90
ceiling event set. These are the loose ends it deliberately did **not** touch.

Read **CLAUDE.md** first; every gate it names applies (Dex-Test-Suite green, verify-provenance
clean, BUILD-MANIFEST + FIXTURE-PROVENANCE updated on re-bundle, fixtures regenerated where math
moves). One node at a time; land each as its own focused PR.

## 1. Port the unification to the 3 OxyDex mean-baseline families STILL on private loops

Task A's §2 table + execution-log discovery scoped the unification to the desat *index/slope/
load/irregularity* family. Three OTHER OxyDex functions still run their **own private
`computeBaselineArr` (trailing-MEAN) loop** and were intentionally left out because they compute
*different metric families* with their own published quintiles/areas and need their own directional
validation:

| Function | Family | Why deferred |
|---|---|---|
| `computeDesaturationProfile` (~L2054) | nadir-duration profile, recovery, `desat.nadir.count`, `dip3Rate`, self-gated artifact events | Feeds `desat` consumed across the app; has artifact self-gating (`selfGateDesat`) the canonical primitive doesn't model. Moving it shifts `nadir.count`, recovery, IEI, O2HR-efficiency, the Integrator emit set. |
| `computeSBII` (~L3478) | SBII (Hui 2024) + quintiles | Published quintile thresholds; ceiling baseline will move SBII and its Q-bin. |
| `computeDesSev` (~L3541) + the duration loop (~L3516) | DesSev (Kulkas) area | Published %·min/hr; ceiling baseline moves the area. |

**Concrete inconsistency this leaves:** `desat.nadir.count` (mean baseline, currently **5** on the
committed fixture) no longer tracks the unified ceiling event set that ODI-4 (**14**) and the Task-A
satellites now share. The Task-A consumers' guards still *gate* on `desat.nadir.count` (e.g.
`computeBreathingIrregularity` requires `desat.nadir.count >= 3`) — harmless today (it's a coarse
"are there any events" check) but worth removing once these families also move to the primitive.

**Required change:** route each through `detectDesatEvents`. `computeDesaturationProfile` is the
delicate one — it needs the **artifact self-gate** preserved: either (a) add an `opts.selfGate`
hook / post-filter step that runs `selfGateDesat` over the primitive's events, or (b) keep
`computeDesaturationProfile` as the one self-gating reference and have it call the SAME primitive for
raw detection, then apply self-gating on top. SBII/DesSev are straightforward (`{dropPct:3,exitPct:3}`
or the ODI-4 set as their current threshold dictates — confirm each against its source loop first).

**Done when:** zero `computeBaselineArr(` calls remain for desat-event *detection* in `oxydex-dsp.js`
(grep clean); SBII/DesSev/nadir-profile validated directionally on the synthetic cohort (no null/NaN
regressions); OxyDex fixture regenerated; gates green. `computeBaselineArr` itself stays in
`oxydex-util.js` (it may still have non-detection uses) — only its *event-detection* call sites go.

## 2. Port the primitive to the other `*-dsp.js` nodes

Task A proved the pattern on OxyDex. The parent brief §5 calls for "the SAME event-unification ported
to the other `*-dsp.js` nodes as separate, identical-shaped passes." Audit each node that does SpO2/
signal desaturation-style event detection (PpgDex, CPAPDex, any future EEGDex) for the same duplicated
rolling-baseline loop and give each its own `detect*Events` primitive + consumer rewire + fixture
regen. Keep each node's primitive node-local (no cross-node runtime dependency, per the parent's §0
prime directive).

## 3. (Optional) Make the OxyDex satellites event-detect ONCE, not per-consumer

Task A's consumers each call `detectDesatEvents` independently (one ceiling-baseline histogram pass
each — ~6 passes over a night). Correct but not minimal. The parent §2.1 suggests memoizing the
baseline/event array on the rows object "for the duration of `processNight`." Cheap win: compute the
ODI-4 event set + ceiling-baseline array ONCE in `processNight` and thread it into the consumers (and
`detectODI`) as `opts.blArr` / a shared events handle. Pure perf; **must not** change any number — lock
it behind the event-for-event test before landing.
