Unified User Profile

One engine, one identity, one panel — across all seven nodes. Just like the export bar, the profile is the same idea reinvented six different ways. This folds it into a single shared module (dex-profile.js), a single saved identity (tepna_profile), and one panel whose only per-node difference is which field groups a node opts into.

scope: OxyDex · HRVDex · PulseDex · ECGDex · PpgDex · GlucoDex · Integrator  ·  pattern parallel to metric-registry.js + the unified export bar

1 The problem — six profiles, three architectures

Every node grew its own profile. Three unrelated code shapes, six localStorage schemas, six DOM-id conventions, and the same formulas copy-pasted (with drift) into each. A person who enters their age in OxyDex re-enters it in every other app — there is no shared identity.

2 Proposed architecture — one engine, opt-in field groups

Mirror metric-registry.js: a single shared module owns the schema, the storage, and every derivation formula. Each node loads it and declares a field-group manifest — the groups it can actually use. The panel, the persistence, and the math come for free; the node only says which fields apply to my signal.

👤 Shared identity tepna_profile

One localStorage record for the person — not per-app. Age/sex/body/fitness entered in any Dex is read by all. Versioned schema; superset of every field. Back-compat shim migrates the six legacy keys on first load.

⚙️ Profile engine dex-profile.js window.DexProfile

The single home for get/save/load, migrate(), and all formulas — Tanaka, expected-rMSSD/RHR norms, Uth–Sørensen VO₂, BMI·BSA·IBW·BMR/RMR·MAP·PP, Karvonen, population norms, altitude & SpO₂ adjustments, A1c↔eAG — with metric as the single stored/computed unit. Renders the panel from a manifest and exposes derive(profile, signalCtx) so every node calls the same math.

4 Field taxonomy — who opts into what

The schema is a superset; each node shows only the groups it can act on. Field-group membership is principled — ECG/PPG omit BP because a single-site signal has no pulse wave (per build brief §6); GlucoDex omits cardio but keeps a light Body group (BMI/waist feed the metabolic & OSA-risk read). All values are stored & computed in metric; an imperial display switch is permissible but metric is the default (per CLAUDE.md).

5 Connecting profile → calculations, and getting more from it

Today most profile fields drive only one node's math, and several collected fields do nothing. With a shared identity the data can cross signals. Each item below is a concrete new use — ordered roughly by value.

A · Correctness gates protect the HR & HRV math. A β-blocker blunts HRmax and resting HR; irregular rhythm / AF breaks RR-interval HRV. Both are single yes/no flags that let the consuming calcs caveat or suppress themselves — Tanaka HRmax, Karvonen zones and VO₂ when a β-blocker is active; rMSSD/SDNN when the rhythm is irregular — instead of reporting confidently-wrong numbers. Correctness, not just precision. touches: derive() validity flags · evidence badge drops to experimental
B · Cross-node resting-HR handoff. OxyDex already derives a robust resting HR from the nocturnal p5 floor. ECG/PPG/Pulse/HRV currently re-derive (worse, from shorter records) or fall back to an age estimate. Persist the best detected resting HR on the shared identity so every node's Karvonen + VO₂ seed from the strongest available source. Same handoff for detected HRmax. touches: prefillFrom(detected) · shared identity
C · BMI as an OSA / metabolic risk prior. Weight + height are collected by every signal node but feed only BMI/BMR display. Obesity is a top-tier OSA risk factor — fold BMI into OxyDex's and ECGDex's apnea-risk band as a documented prior (raises pre-test probability, never fabricates events). Surfaces a number we already have. touches: apneaRisk() context · evidence badge: heuristic
D · Diabetes status crosses into the autonomic nodes. GlucoDex collects diabetes status & therapy; diabetes elevates both OSA and cardiac-autonomic-neuropathy risk. When the Integrator fuses a CGM with ECG/HRV, surface a context flag (e.g. "low HRV in the presence of diabetes — consider CAN") rather than leaving the fact siloed. touches: Integrator fusion notes · shared identity carries metabolic group
E · Elevation honesty for HRV norms. OxyDex adjusts SpO₂ thresholds for altitude; the HRV nodes only scale VO₂ — yet their rMSSD/resting-HR norms are sea-level. Since elevation is already on the identity, add a one-line caveat to the HRV read above ~2500 m (acute altitude raises resting HR / lowers HRV). touches: renderHero note · no new field
F · Generalise the ground-truth check. GlucoDex's "lab A1c validates GMI" is the same pattern as "measured VO₂max validates the estimate." Make ground-truth validation a first-class engine concept: any node with a measured truth (A1c, VO₂, a cuff BP) shows the device-vs-truth delta in one consistent chip. touches: derive() returns groundTruthChecks[]
G · Body composition → a real energy budget. Body-fat % switches RMR to Katch-McArdle (lean-mass based, beats Mifflin for lean/obese); activity level turns RMR into TDEE (daily burn); waist gives waist-to-height ratio (visceral / OSA prior). All three reuse fields the body group already holds. touches: derive() → rmr / tdee / whtr · badges: validated / heuristic

7 Open questions for you

  1. Shared identity scope. Confirmed: one record across all Dexes, and it works for a single-Dex user too — no value assumes a particular Dex entered it. Any field a user never touches stays on its cited population default.
  2. Group membership — confirmed. CPAP added to HRVDex; a light Body group (BMI/waist) added to GlucoDex for the metabolic/OSA prior; β-blocker & AF correctness flags join Cardio on every HRV-bearing node.
  3. Cross-signal scope. Ship the safe wins first — B (resting-HR handoff) and A (correctness gates) — then the clinical-context priors C (BMI→OSA) and D (diabetes→CAN) as a reviewed second pass.
  4. Units — settled. Metric is canonical and the default everywhere; an imperial display switch is permissible (display layer only — stored/computed values stay metric). Now in CLAUDE.md.
  5. Rollout. One re-bundle-all pass (cleanest, heaviest), or land the engine first on the three IIFE nodes (ECG/PPG/Gluco — least risk), then migrate OxyDex/HRVDex/PulseDex in a second pass?
